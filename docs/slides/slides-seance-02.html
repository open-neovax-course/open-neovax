Le contrat get_score

La seule fonction que le pipeline appelle

La signature

def get_score(candidate: Candidate) -> tuple[str, float]:
    return ("score_name", 0.42)

Entrée : candidate

.peptide_mutSéquence mutante
.peptide_wtSéquence wild-type
.mut_pos_1basedPosition de la mutation
.geneGène source
.hla_alleleAllèle HLA

Sortie : tuple[str, float]

  • str — nom unique de votre score
  • float — valeur calculée
  • Pas de NaN, pas d'inf
  • Pas de None

Ce qui se passe quand le pipeline tourne

1. Découverte Le pipeline scanne modules/ et trouve votre fichier groupe_XX.py
2. Import Il importe votre module. Si l'import crashe → warning + module ignoré
3. Appel Pour chaque candidat, il appelle get_score(candidate)
4. Validation Il vérifie : tuple de 2 ? str + float ? pas NaN ? Sinon → warning + ignoré
5. Stockage Le score est stocké dans candidate.scores["votre_nom"]

Votre module ne peut pas crasher le pipeline. Le pire qui arrive : un warning et votre score est ignoré.

Normalisation & nommage

Les règles du jeu

La normalisation : on s'en occupe

Le core applique une normalisation min-max sur chaque score :

score_norm = (score - min) / (max - min)

→ Résultat toujours entre 0 et 1
  • Pas besoin de normaliser vous-mêmes
  • Vos scores bruts peuvent être dans n'importe quelle plage
  • Si tous les candidats ont le même score → 0.5

Attention au signe !

Après normalisation, 1 = le meilleur, 0 = le pire.

Donc un score élevé doit signifier "bon candidat".

Hydrophobicité équilibrée → score élevé

Self-match trouvé → score négatif (pénalité)

Ancre P2 = D → score négatif (pire ancre)

Nommage du score

Convention : <département>_<concept>[_détail]

Bons noms

A_hydrophobicity_kdDept A, Kyte-Doolittle
A_net_chargeDept A, charge nette
B_proteasome_ctermDept B, C-terminal
C_hla_anchor_p2Dept C, ancre position 2
D_self_exact_matchDept D, match exact

Mauvais noms

scoreTrop vague
my_scoreNon descriptif
resultRien à voir
hydroPas le département
Si deux modules retournent le même nom → écrasement silencieux. Le dernier gagne.

Par département

Pistes pour votre score naïf

Dept A — Propriétés physico-chimiques

Vous travaillez directement sur la séquence candidate.peptide_mut

ModuleScore naïf
A1Moyenne Kyte-Doolittle sur le peptide
A2Compter D, E (−) et K, R, H (+)
A3Entropie de Shannon sur la composition AA
A4Diff. d'une propriété entre WT et MUT
Exemple A1 : 
KD = {"A": 1.8, "R": -4.5,
      "N": -3.5, "D": -3.5,
      "I": 4.5, "L": 3.8,
      "V": 4.2, "M": 1.9, ...}

def get_score(candidate):
    pep = candidate.peptide_mut
    avg = sum(KD.get(aa, 0.0)
              for aa in pep) / len(pep)
    return ("A_hydrophobicity_kd", avg)

Dept B — Processing & présentation

Vous modélisez les étapes intracellulaires (protéasome, TAP, ERAP)

ModuleScore naïf
B1Dernier AA du peptide : favorable ou pas ? (table de lookup)
B2Score basé sur le C-terminal + composition globale
B3Premier AA du peptide : résistant au trimming ?
B4Longueur OK ? Caractères valides ? WT ≠ MUT ?
Exemple B4 : 
VALID_AA = set("ACDEFGHIKLMNPQRSTVWY")

def get_score(candidate):
    pep = candidate.peptide_mut
    score = 1.0  # on part de 1 (OK)

    if not (8 <= len(pep) <= 11):
        score -= 0.5
    if not all(aa in VALID_AA for aa in pep):
        score -= 0.5

    return ("B_sanity_check", score)

Dept C — Liaison HLA

Vous utilisez la matrice PSSM (data/hla_pssm_A0201.csv)

ModuleScore naïf
C1Valeur PSSM à la position P2
C2Valeur PSSM à la position P9 (dernière)
C3Somme PSSM sur toutes les positions
C4Score PSSM(MUT) − Score PSSM(WT)
La PSSM : matrice 20 AA × 9 positions.
Chaque case = score log-odds.
Positif = favorable, négatif = défavorable.
Exemple C1 : 
# Charger la PSSM (version naïve)
import pandas as pd
pssm = pd.read_csv(DATA_DIR / "hla_pssm_A0201.csv",
                    index_col="aa")

def get_score(candidate):
    pep = candidate.peptide_mut
    p2_aa = pep[1]  # P2 = index 1
    score = pssm.loc[p2_aa, "P2"]
    return ("C_hla_anchor_p2", score)

Dept D — Sécurité & soi

Vous comparez avec le corpus self (data/human_peptides_small.txt)

ModuleScore naïf
D1Le peptide MUT est-il dans le corpus ? Oui → grosse pénalité
D2Distance de Hamming minimale avec le corpus
D3mut_pos dans [1, len(pep)] ? WT ≠ MUT ?
Exemple D1 : 
# Charger le corpus self
with open(DATA_DIR / "human_peptides_small.txt") as f:
    SELF = set(line.strip() for line in f)

def get_score(candidate):
    pep = candidate.peptide_mut
    if pep in SELF:
        return ("D_self_exact_match", -10.0)
    return ("D_self_exact_match", 0.0)

Score négatif = pénalité.
Après normalisation, self-match → score 0.

Implémentation

2h30 — À vous de jouer !

Partir du template

Votre fichier modules/groupe_XX.py a déjà la bonne structure :

# ══════════════════════════════════════
#  CONSTANTES
# ══════════════════════════════════════
DATA_DIR = Path(__file__).resolve().parent.parent / "data"
SCORE_NAME = "template_score"  ← à changer !


# ══════════════════════════════════════
#  FONCTIONS INTERNES (privées)
# ══════════════════════════════════════
def _compute_something(peptide: str) -> float:
    ...  ← votre logique ici


# ══════════════════════════════════════
#  FONCTION PUBLIQUE
# ══════════════════════════════════════
def get_score(candidate: "Candidate") -> tuple[str, float]:
    peptide = candidate.peptide_mut
    score_value = _compute_something(peptide)
    return (SCORE_NAME, score_value)

Checklist pendant l'implémentation

À faire

  • Changer SCORE_NAME (convention dept_concept)
  • Écrire votre logique dans _compute_something
  • Tester manuellement avec Streamlit
  • Vérifier le signe : élevé = bon candidat

Erreurs fréquentes

  • Oublier de changer SCORE_NAME
  • Diviser par len(pep) quand pep = ""
  • Score inversé (self-match → score positif)
  • Crash sur un AA inconnu (X, *)

Pour tester en live : streamlit run app.py → cochez "Demo file" → "Run analysis"
Vous devriez voir votre colonne de score apparaître !

Bon code vs. mauvais code

Mauvais : 
def get_score(candidate):
    p = candidate.peptide_mut
    s = 0
    for i in range(len(p)):
        if p[i] == "L":
            s += 2
        elif p[i] == "V":
            s += 1.5
    return ("score", s)
  • Nom "score" non descriptif
  • Magic numbers (2, 1.5)
  • Variables cryptiques (p, s, i)
Bon : 
SCORE_NAME = "C_hla_anchor_p2"
ANCHOR_SCORES = {"L": 2.0, "M": 1.5,
                 "V": 1.0, "A": -0.5}
DEFAULT_SCORE = -1.0

def get_score(candidate):
    peptide = candidate.peptide_mut
    p2_aa = peptide[1]
    score = ANCHOR_SCORES.get(
        p2_aa, DEFAULT_SCORE)
    return (SCORE_NAME, score)
  • Nom explicite
  • Constantes nommées
  • Variables lisibles

Premier test

Vérifier que ça marche (et que ça marchera demain)

Votre premier test

Créez tests/test_groupe_XX.py :

from logic.types import Candidate
from modules.groupe_XX import get_score


def _make_candidate(peptide_mut="SLMAFTIAV"):
    return Candidate(
        candidate_id="TEST",
        peptide_wt="SSMAFTIAV",
        peptide_mut=peptide_mut,
        mut_pos_1based=2,
    )


def test_returns_valid_tuple():
    """get_score returns a (str, float) tuple."""
    name, value = get_score(_make_candidate())
    assert isinstance(name, str)
    assert isinstance(value, float)
    assert len(name) > 0

Pourquoi écrire des tests ?

🛡️

Sécurité

Vous modifiez votre code → les tests vérifient que rien n'est cassé

📋

Documentation

Un test montre comment votre fonction est censée être utilisée

🤖

CI automatique

Vos tests tournent à chaque push — la CI vous protège

Aujourd'hui : 1 test minimum. En séance 4 : 3 tests minimum (nominal + edge case + invalid).

Avant de partir

# 1. Vérifier que le test passe
pytest tests/test_groupe_XX.py -v

# 2. Vérifier le style
ruff check modules/groupe_XX.py
black modules/groupe_XX.py

# 3. Commiter et pousser
git add modules/groupe_XX.py tests/test_groupe_XX.py
git commit -m "feat: naive score implementation"
git push

# 4. Vérifier la CI sur GitHub
# → votre PR doit être verte
